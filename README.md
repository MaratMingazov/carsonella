# Carsonella

Kotlin Multiplatform + Compose Multiplatform приложение-симулятор физики и химии. Пользователь видит «вселенную» на Canvas, перетаскивает из палитры частицы (фотон, электрон, протон, водород, кислород, H₂, звезда, SpaceModule) и наблюдает реакции в реальном времени.

## Цель проекта

1. **Изучить, как устроен мир.** Как из элементарных частиц образуются атомы, из атомов — молекулы, из молекул — более сложные соединения, и как в итоге это приводит к появлению жизни и эволюции.
2. **Воплотить это в виде небольшой игры-стратегии.** С одной стороны эволюция идёт своим чередом — можно просто наблюдать. С другой — игрок может вмешиваться в этот мир (пока не решено как именно: возможно копить ресурсы, строить какие-то объекты) и направлять ход эволюции.

## Архитектура домена (`shared/.../chemistry/`)

### Сущности

`Entity<State>` — общий интерфейс. Поведения подмешиваются через делегирование (`behavior/*.kt`):

- `DeathNotifiable` — сообщить миру о смерти
- `NeighborsAware` — получать актуальный список соседей
- `ReactionRequester` — отправлять запрос на реакцию в мир
- `EnvironmentAware` — знать своё окружение
- `LogWritable` — писать в общий лог

Конкретные классы: `SubAtom`, `Atom`, `Molecule`, `Star`, `SpaceModule`, `RecombinationModule`. Каждая сущность — корутина с циклом `init()` + `delay(10)`, который двигает её, считает силы и при необходимости шлёт `requestReaction(...)`.

### Элементы

`Element` (enum) + `Details` — большая таблица элементов с параметрами:

- масса, число электронов/протонов/нейтронов, радиус
- `energyLevels` — энергетические уровни (для фотоионизации и спонтанной эмиссии)
- `ion` — ион, который получится при потере электрона
- `recombinationElement` — что получится при захвате электрона
- `alphaReactionResult` — продукт альфа-захвата (нуклеосинтез в звёздах)
- `energyBondDissociation` / `dissociationElements` — для фотодиссоциации молекул

### Окружение

`IEnvironment` — у каждой частицы есть «среда» (центр, радиус, `TemperatureMode.Space | Star`, дети). Любая частица сама может быть средой для других: звезда «содержит» свои протоны/электроны, и альфа-синтез работает только внутри звезды.

### Реакции (`chemistry/chemical_reaction/`)

`ChemicalReactionResolver` хранит список `ReactionRule` и для каждого запроса выбирает правило с максимальным `weight()` (рандом среди равных по весу).

Реализованные правила:

- `PhotoIonization` — отрыв электрона фотоном
- `PhotoDissociation` — расщепление молекулы фотоном
- `SpontaneousEmission` — возбуждённый атом излучает фотон
- `StarEmission` — звезда излучает частицы в космос
- `RecombinationReaction` — ион + электрон → атом + фотон
- `StarAlphaReaction` — альфа-захват в недрах звёзд (нуклеосинтез)
- `AtomPlusAtomToMolecule` — параметризованное правило для синтеза (p+p→D, D+p→³He, C+C→Ne, H+H→H₂, O+O→O₂, O+H₂→H₂O и т. д.)

Каждое правило возвращает `ReactionOutcome { consumed, spawn, updateState, description }`.

## Игровой цикл (`composeApp/.../world/`)

`World` владеет:

- `entities` (`SnapshotStateList<Entity<*>>`)
- палитрой элементов
- корневым `Environment` (радиус 10000)
- `Channel<ReactionRequest>` для реакций
- `EntityGenerator`, `ChemicalReactionResolver`

Запрос реакции идёт по цепочке:

```
Entity.requestReaction(reagents)
    → Channel<ReactionRequest>
    → World.runReaction (под mutex)
    → ChemicalReactionResolver.resolve
    → consumed.destroy() + spawn() + updateState()
```

## UI

- `App.kt` — корень: `Row { LeftPanel | RightPanel }`.
- `LeftPanel` — палитра элементов (drag source) + панель выбранной сущности.
- `RightPanel` — Canvas с миром:
  - hit-test, hover/select частиц
  - WASD / стрелки двигают выбранную частицу импульсом
  - Space «стреляет» копией выделенной частицы в направлении курсора
  - снизу — консоль логов реакций
- `EntityRenderer` диспетчеризует отрисовку по типу состояния; звезда — пульсирующий радиальный градиент.
- `DragAndDrop.kt` — дроп с палитры на канву → `entityGenerator.createEntity(...)`.

## Технические концерны (TODO / на будущее)

Зафиксировано, чтобы не забыть. Список по убыванию приоритета:

1. **Симуляция: единый tick на World вместо корутины-на-сущность.**
   Сейчас каждая Entity — своя корутина с `delay(10)`. Работает на десятках, но при сотнях/тысячах timing поплывёт, а порядок шагов между сущностями станет недетерминированным. Здоровее: один tick-loop в `World`, который шагает всех синхронно. Это про производительность и про воспроизводимость.

2. **`getNeighbors()` сейчас O(N²).**
   На каждом шаге каждой сущности фильтруется весь список соседей. Файл `Grid2D.kt` уже лежит — видимо, в процессе перехода на spatial hashing. Без него масштабирование закончится быстро.

3. **`weight()` у всех правил возвращает `0f`.**
   Резолвер из-за этого выбирает случайное правило среди подходящих. Задумка «приоритизация реакций по вероятности» пока не работает — её нужно либо запустить, либо честно признать, что выбор просто случайный.

4. **`Entity` дефолтно бросает `Not Supported` на методы среды.**
   Type-smell: не каждая Entity — среда, и интерфейс это знает. Лучше отдельный `EnvironmentProvider` интерфейс, реализуемый только теми, кто реально среда (`Star`, `SpaceModule`).

5. **`Element` как enum + большой `Details`-каталог.**
   Сейчас норм, но когда дойдёт до органики (десятки/сотни молекул), enum станет узким горлом. Держать в уме переход на `data class Element` + регистр.

6. **Глобальный `random = Random(1)` + кое-где `Random.nextFloat()` без сида.**
   Если когда-нибудь понадобится воспроизводимость симуляции (а для отладки она нужна) — это придётся переделывать. Лучше передавать `Random` через `World`.

7. **`EntityState` с `var`-полями, но используется как immutable через `copyWith`.**
   Одно или другое — иначе путает читателя. Лучше сделать поля `val`.

8. **Мелочи:**
   - `id = ""` у нескольких правил, `id = "PhotoIonization"` у двух разных правил (`PhotoIonization` и `StarEmission`).
   - Смешение стилей в `Element` enum: `PHOTON` (всекапс) vs `Proton` / `Star` / `Ni` (CamelCase / короткое).

9. **Пустой `server` модуль.**
   Если планируется мультиплеер/сохранения — это отдельный большой проект. Если нет — можно убрать из `settings.gradle.kts`, чтобы не мозолил.

## Стратегические заметки (геймплей)

1. **Определить «победу» / прогрессию.**
   Что для игрока — конец партии? Достижение жизни? Технологическое дерево? Колонизация одной звёздной системы? Это направит, какие механики достраивать, а какие — нет.

2. **Разнести симуляцию на эпохи.**
   Между «протон поймал электрон» и «возникла самовоспроизводящаяся молекула» в реальной вселенной 10⁹–10¹⁷ разницы по времени и масштабу — на одной модели всю цепочку не покрыть. Естественные эпохи: космогенез → нуклеосинтез в звёздах → планетарная химия → пребиотика → жизнь. Внутри одной эпохи модель остаётся простой; переходы между эпохами — отдельная игровая веха.

3. **«Реакторы» как игровая механика, обходящая медленные естественные процессы.**
   `SpaceModule` / `RecombinationModule` — уже шаг в эту сторону. Игрок строит «ускорители», которые сжимают невероятно медленные естественные процессы. Это рабочая формула в духе Factorio / Oxygen Not Included, её стоит развивать.

4. **Перевести симуляцию на единый tick до того, как обрастёт частицами.**
   Потом будет больно (см. п. 1 в технических концернах).
