# Carsonella

Kotlin Multiplatform + Compose Multiplatform приложение-симулятор физики и химии. Пользователь видит «вселенную» на Canvas, перетаскивает из палитры частицы (фотон, электрон, протон, водород, кислород, H₂, звезда, SpaceModule) и наблюдает реакции в реальном времени.

## Цель проекта

1. **Изучить, как устроен мир.** Как из элементарных частиц образуются атомы, из атомов — молекулы, из молекул — более сложные соединения, и как в итоге это приводит к появлению жизни и эволюции.
2. **Воплотить это в виде небольшой игры-стратегии.** С одной стороны эволюция идёт своим чередом — можно просто наблюдать. С другой — игрок может вмешиваться в этот мир (пока не решено как именно: возможно копить ресурсы, строить какие-то объекты) и направлять ход эволюции.

## Визуальный стиль и настроение

Целевая эстетика — **созерцательная, расслабляющая, «медитативный микромир»** в духе *Osmos* и *Eufloria*. Игра, в которую играют не спеша, любуясь происходящим.

- Тёмный градиентный фон, ощущение бесконечного космоса.
- Частицы — полупрозрачные светящиеся сферы с мягким halo (радиальные градиенты, аддитивный blend, лёгкий bloom).
- За движущимися частицами тянется затухающий световой след (trails).
- Звёздные реакции — не «вспышка экшена», а тихий всплеск света.
- Минимум UI на канве, ambient-музыка / тишина, никакого хардкора и спешки.

Опционально (космическая фаза): реалистичные цвета звёзд по температуре и туманности на фоне в духе снимков Hubble/JWST.

## Архитектура домена (`shared/.../chemistry/`)

### Сущности

`Entity<State>` — общий интерфейс. Поведения подмешиваются через делегирование (`behavior/*.kt`):

- `DeathNotifiable` — сообщить миру о смерти
- `NeighborsAware` — получать актуальный список соседей
- `ReactionRequester` — отправлять запрос на реакцию в мир
- `EnvironmentAware` — знать своё окружение
- `LogWritable` — писать в общий лог

Конкретные классы: `SubAtom`, `Atom`, `Molecule`, `Star`, `SpaceModule`, `RecombinationModule`. У каждой синхронный `step()`, который вызывается миром раз за tick: двигает сущность, считает силы, при необходимости шлёт `requestReaction(...)`. Никаких собственных корутин или `delay` — мир — единственное «время».

### Элементы

`Element` (enum) + `Details` — большая таблица элементов с параметрами:

- масса, число электронов/протонов/нейтронов, радиус
- `energyLevels` — энергетические уровни (для фотоионизации и спонтанной эмиссии)
- `ion` — ион, который получится при потере электрона
- `recombinationElement` — что получится при захвате электрона
- `alphaReactionResult` — продукт альфа-захвата (нуклеосинтез в звёздах)
- `energyBondDissociation` / `dissociationElements` — для фотодиссоциации молекул

### Окружение

`IEnvironment` — у каждой частицы есть «среда» (центр, радиус, `TemperatureMode.Space | Star`, дети). Любая частица сама может быть средой для других: звезда «содержит» свои протоны/электроны, и альфа-синтез работает только внутри звезды.

### Реакции (`chemistry/chemical_reaction/`)

`ChemicalReactionResolver` хранит список `ReactionRule` и для каждого запроса выбирает правило с максимальным `weight()` (рандом среди равных по весу).

Реализованные правила:

- `PhotoIonization` — отрыв электрона фотоном
- `PhotoDissociation` — расщепление молекулы фотоном
- `SpontaneousEmission` — возбуждённый атом излучает фотон
- `StarEmission` — звезда излучает частицы в космос
- `RecombinationReaction` — ион + электрон → атом + фотон
- `StarAlphaReaction` — альфа-захват в недрах звёзд (нуклеосинтез)
- `StarPPChain` — pp-цепочка в звезде. **pp-I**: p+p→D⁺, D⁺+p→³He²⁺, ³He²⁺+³He²⁺→⁴He²⁺+2p. **pp-II** (через литий): ⁷Be⁴⁺+e⁻→⁷Li³⁺ (electron capture), ⁷Li³⁺+p→2 ⁴He²⁺ (горение лития). Стартовый шаг pp-II (³He²⁺+⁴He²⁺→⁷Be⁴⁺) живёт в `StarAlphaReaction` — через `alphaReactionResult` у ³He²⁺.
- `StarCarbonBurning` — горение углерода: ¹²C+¹²C → ²⁰Ne+⁴He / ²³Na+p / ²⁴Mg (канал выбирается случайно)
- `StarOxygenBurning` — горение кислорода: ¹⁶O+¹⁶O → ²⁸Si+⁴He / ³¹P+p / ³¹S+n (канал выбирается случайно)
- `AtomPlusAtomToMolecule` — параметризованное правило для молекулообразования (D⁺+e⁻→D, H+H→H₂, O+O→O₂, O+H₂→H₂O)

Каждое правило возвращает `ReactionOutcome { consumed, spawn, updateState, description }`.

### Что моделируется сейчас (физическая цепочка)

```
звезда (StarEmission)
    → нуклеосинтез внутри звезды:
        · pp-I (StarPPChain): p+p→D⁺, D⁺+p→³He²⁺, ³He²⁺+³He²⁺→⁴He²⁺+2p
        · pp-II (StarAlphaReaction + StarPPChain): ³He²⁺+⁴He²⁺→⁷Be⁴⁺, ⁷Be⁴⁺+e⁻→⁷Li³⁺, ⁷Li³⁺+p→2 ⁴He²⁺
        · альфа-захват (StarAlphaReaction): ⁴He→⁸Be→¹²C→¹⁶O→…→⁵⁶Ni
        · горение углерода (StarCarbonBurning): ¹²C+¹²C → ²⁰Ne/²³Na/²⁴Mg
        · горение кислорода (StarOxygenBurning): ¹⁶O+¹⁶O → ²⁸Si/³¹P/³¹S
    → излучение в космос (StarEmission выбрасывает p⁺/e⁻/O⁸⁺ наружу)
    → фотоионизация / рекомбинация (PhotoIonization, RecombinationReaction, SpontaneousEmission)
    → молекулообразование (AtomPlusAtomToMolecule: H+H→H₂, O+O→O₂, O+H₂→H₂O, …)
    → распад молекул фотонами (PhotoDissociation)
```

Это уже физически довольно правдоподобная база — от субатомных частиц через звёздный нуклеосинтез до простых молекул.

### Маппинг стадий цепочки на правила

| # | Стадия | Реакция | Что делает |
|---|---|---|---|
| 0 | **Спавн звезды** | — | `World.start()` создаёт `Star` как `Entity`. Звезда — не реакция, а сущность-среда. |
| 1 | **Звезда работает (топливо)** | `StarEmission` (ветка генерации) | При `< 10` детях создаёт внутри звезды свежий p⁺ или e⁻ |
| 2 | **Нуклеосинтез: pp-I** | `StarPPChain` | p+p→D⁺, D⁺+p→³He²⁺, ³He²⁺+³He²⁺→⁴He²⁺+2p |
| 2 | **Нуклеосинтез: pp-II** | `StarPPChain` + `StarAlphaReaction` | ³He²⁺+⁴He²⁺→⁷Be⁴⁺ (через `alphaReactionResult`), ⁷Be⁴⁺+e⁻→⁷Li³⁺, ⁷Li³⁺+p→2 ⁴He²⁺ |
| 2 | **Нуклеосинтез: горение углерода** | `StarCarbonBurning` | ¹²C+¹²C → ²⁰Ne+⁴He / ²³Na+p / ²⁴Mg (случайный канал) |
| 2 | **Нуклеосинтез: горение кислорода** | `StarOxygenBurning` | ¹⁶O+¹⁶O → ²⁸Si+⁴He / ³¹P+p / ³¹S+n (случайный канал) |
| 2 | **Нуклеосинтез: альфа-захват** | `StarAlphaReaction` | ⁴He+X→Y по цепочке ⁴He→⁸Be→¹²C→¹⁶O→²⁰Ne→…→⁵⁶Ni |
| 3 | **Излучение в космос** | `StarEmission` (ветка выброса) | При `≥ 10` детях выкидывает p⁺/e⁻/O⁸⁺ за пределы звезды (через `updateMyEnvironment`) |
| 4 | **Ионизация** | `PhotoIonization` | атом + γ → ион + e⁻ (или поглощение энергии, если порог не пройден) |
| 4 | **Рекомбинация** | `RecombinationReaction` | ион + e⁻ → атом + γ |
| 4 | **Релаксация (люминесценция)** | `SpontaneousEmission` | возбуждённый атом сам излучает фотон, спускаясь на нижний энергоуровень |
| 5 | **Молекулообразование** | `AtomPlusAtomToMolecule` | D⁺+e⁻→D, H+H→H₂, O+O→O₂, O+H₂→H₂O |
| 5* | **Распад молекул (обратное)** | `PhotoDissociation` | молекула + γ → 2 атома |

**Заметки:**

- **Звёздный нуклеосинтез разнесён по специализированным правилам.** Раньше всё (pp-цепочка, горение C, горение O) ехало на `AtomPlusAtomToMolecule` в Star-режиме. Сейчас каждая ветвь — отдельный `Star…`-класс со своей логикой выбора канала. `AtomPlusAtomToMolecule` остался только для молекулообразования (4 инстанса) — звёздных реакций он больше не делает, хоть параметр `temperatureMode` в нём пока сохранён.
- **`StarCarbonBurning` / `StarOxygenBurning` — рандомизация внутри правила.** У реакции три возможных канала (для C: ²⁰Ne+⁴He / ²³Na+p / ²⁴Mg, для O: ²⁸Si+⁴He / ³¹P+p / ³¹S+n) — конкретный выбирается в `matches()` через `entityGenerator.random`. Это эквивалент прежнего random-выбора между тремя `AtomPlusAtomToMolecule` с равным весом, но компактнее.
- **`StarEmission` совмещает две роли** через `if/else` (генерация vs выброс). Можно при желании разнести на `StarFuelGeneration` + `StarOutflow`.
- **Стадия 4 — внутренний цикл**: атом ↔ возбуждённый атом ↔ ион. `PhotoIonization` (поглощение) ↔ `SpontaneousEmission` (сброс) и `PhotoIonization` (отрыв) ↔ `RecombinationReaction`. В консоли логов это видно как постоянное «туда-сюда».

### Чего из реальной физики ещё нет

- **Литий (⁷Li): pp-II реализована, но игроку Li пока не виден.** ✅ Производство через pp-II работает: ³He²⁺+⁴He²⁺→⁷Be⁴⁺ (`StarAlphaReaction`), ⁷Be⁴⁺+e⁻→⁷Li³⁺ (electron capture, нейтрино опущено), ⁷Li³⁺+p→2 ⁴He²⁺ (горение лития обратно в гелий). Литий рождается внутри звезды, но без обобщения `StarEmission` (см. следующий пункт) наружу не выходит и в палитре не появляется. Альтернативные источники, которые в проекте пока не реализованы:
  1. **Первичный нуклеосинтез (BBN).** Почти весь ⁷Li во Вселенной (~10⁻¹⁰ от H) образовался сразу после Большого Взрыва — это **главный** природный источник. Реализация: отдельное правило `BigBangNucleosynthesis`, стартующее в холодной космической среде до зажигания звёзд. Требует ввести понятие «эпох» симуляции (сейчас всё происходит одновременно).
  2. **Космические лучи (spallation).** Высокоэнергичные протоны разбивают межзвёздные ядра C/N/O на Li/Be/B. Реализация: отдельное правило `CosmicRaySpallation`. Имеет смысл подключать только когда в космосе уже накопились тяжёлые ядра — то есть на поздней стадии.

- **Бор (¹⁰B, ¹¹B) и бериллий-9 (⁹Be) — нет вообще никакого пути производства.** В отличие от лития, у бора и ⁹Be нет звёздного источника: дыры A=5 и A=8 в карте стабильных ядер звёздная α-цепочка перепрыгивает через тройной α-процесс (3⁴He→¹²C), минуя Li/Be/B. У лития есть обходной путь через pp-II (его мы реализовали), у бора — нет. Хуже того: любой бор, попавший в звезду, мгновенно сгорает в протонном фоне (¹⁰B+p→⁷Be+α при ~5 МК, ¹¹B+p→3α) — поэтому в Солнце Li/Be/B в ~10⁹ раз реже C/N/O. Единственный реальный источник Li/Be/B во Вселенной (Reeves–Fowler–Hoyle, 1970) — **космические лучи (spallation)**: ГэВ-протоны разбивают межзвёздные C/N/O на лёгкие фрагменты. Минорный канал — ν-процесс в сверхновых (~10% ¹¹B), но это уже другая эпоха. Реализация: то же правило `CosmicRaySpallation` из пункта 2 TODO про литий — когда дойдём до него, имеет смысл добавить сразу всё семейство (⁹Be, ¹⁰B, ¹¹B, плюс ⁶Li как доп. канал к существующему ⁷Li). До того момента **бор-элементов в `Entity` не заводить** — не хотим декорации без реакций.

- **Азот (¹⁴N) и CNO-цикл — отсутствует весь путь производства нечётных-Z элементов.** Альфа-цепочка (`StarAlphaReaction`) идёт только по чётным Z: He(2) → C(6) → O(8) → Ne(10) → Mg(12) → Si(14) → … Все нечётные (N, Na, Al, P, Cl, K) — продукты других процессов. Главный путь для азота — **CNO-цикл (Бете, 1938)**: уже существующий ¹²C служит катализатором горения водорода, цикл выходит на равновесие, в котором ¹⁴N накапливается (т.к. шаг ¹⁴N+p — узкое горлышко). Это объясняет, почему N в межзвёздной среде так много после нескольких поколений звёзд. Решено идти по **полному пути с честным β⁺-распадом** — это откроет дорогу к остальной β-управляемой физике (s/r-процесс, нейтронизация).

  Что нужно реализовать:
  1. ✅ **Позитрон** как новый `SubAtom` (`POSITRON` / e⁺) — сделано. `mass=0.1`, `e=0, p=1, n=0` (`p=1` тут маркер положительного единичного заряда, а не «содержит протон» — см. комментарий в `Entity.kt`). В `SubAtom.step()` отдельный `initPositron` (копия `initProton`: движение + `requestReaction` для будущей аннигиляции). В `SubAtomRenderer` электрон обновлён до синего круга с белым «−», позитрон — красный круг с белым «+».
  2. ✅ **Изотопы для CNO** — сделано. Добавлены ¹³C⁶⁺, ¹³N⁷⁺, ¹⁴N⁷⁺, ¹⁵N⁷⁺, ¹⁵O⁸⁺ (только полностью ионизованные формы — нейтралы и промежуточные ионы появятся, если/когда понадобится выпускать N в космос).
  3. ✅ **Спонтанный β⁺-распад** — сделано как generic `BetaPlusDecay` (single-reagent rule по образцу `SpontaneousEmission`, инфраструктура уже была). Data-driven через `Details.betaPlusDecayResult`. Сейчас работает ¹³N→¹³C+e⁺ и ¹⁵O→¹⁵N+e⁺ с 2%-шансом за тик (~1 сек среднее). См. umbrella-секцию «Радиоактивный распад» ниже.
  4. **Правило `StarCNOCycle`** в стиле `StarPPChain` (when по первому реагенту), четыре протон-захватных шага: ¹²C+p→¹³N, ¹³C+p→¹⁴N, ¹⁴N+p→¹⁵O, ¹⁵N+p→¹²C+⁴He. Работает только в `TemperatureMode.Star`.
  5. **Правило `Annihilation`** — `e⁻ + e⁺ → 2γ` (по 511 кэВ каждый фотон). **Обязательно вместе с β⁺-распадом**, иначе позитроны, рождённые в шаге 3, будут накапливаться в мире вечно. По образцу `RecombinationReaction`: бинарная реакция, применима в любой среде (не только в звезде). Это первый случай аннигиляции в проекте — в будущем можно расширить и на другие пары частица/античастица, если они появятся.

  «Затравка» для запуска цикла приходит бесплатно: ¹²C уже образуется в `StarAlphaReaction` из ⁸Be+α. Бонус — инфраструктура β⁺-распада из пункта 3 переиспользуется для других нестабильных изотопов: тех же ⁸Be→2⁴He (сейчас обходится в `StarAlphaReaction` через `alphaReactionResult` без явного распада), будущие нейтрон-богатые ядра, расширения для CNO-II/III.

- **`StarEmission` должна выбрасывать любой элемент, а не только p⁺/e⁻/O⁸⁺.** Сейчас в ветке выброса хардкод трёх типов частиц. Но внутри звезды нуклеосинтез производит десятки элементов (через `StarAlphaReaction`, `StarPPChain`, `StarCarbonBurning`, `StarOxygenBurning` — вплоть до ⁵⁶Ni), и все они «застревают» в звезде, наружу выходят только три. Нужно: выбирать выбрасываемый элемент случайным образом из текущих children звезды. Прямо связано с pp-II/литием — без этого ⁷Li, даже если его произвести в звезде, никогда не появится в космосе и игрок его не увидит. После этого пункт 1 «реализовать pp-II в `StarPPChain`» становится играбельным.

- **Радиоактивный распад — зонтичный термин** для нескольких разных ядерных процессов. Часто их путают, но это разные механизмы — будем добавлять по мере появления соответствующих элементов:
  - **α-распад** — ядро выбрасывает ⁴He (Z → Z−2, A → A−4). Тяжёлые ядра: U, Th, Ra, Rn, Po. У нас **обратный** процесс (α-захват) уже есть — `StarAlphaReaction`. Прямой «выброс α» — отдельное правило, понадобится для урановой цепочки (²³⁸U → ²³⁴Th → … → ²⁰⁶Pb, ~14 шагов через чередование α и β⁻).
  - **β⁻-распад** — `n → p + e⁻ + ν̄ₑ` (Z → Z+1, A та же). Нейтрон-избыточные ядра: ¹⁴C, ³H, ⁹⁰Sr, продукты r/s-процесса и деления. У нас сейчас нет нейтрон-избыточных изотопов, правило не нужно. Симметрично β⁺ — то же generic-правило с полем `betaMinusDecayResult` в Details и спавном электрона вместо позитрона.
  - **β⁺-распад** — `p → n + e⁺ + νₑ` (Z → Z−1, A та же). ✅ **Реализовано** как generic `BetaPlusDecay` (data-driven через `Details.betaPlusDecayResult`). Сейчас работает для ¹³N→¹³C и ¹⁵O→¹⁵N (шаги CNO-цикла).
  - **Electron capture (EC)** — `p + e⁻ → n + νₑ`. Конкурирует с β⁺ при низком Q-value (когда не хватает энергии на пару e⁺/e⁻, порог ~1.022 МэВ). ✅ **Реализовано** как кейс в `StarPPChain` для ⁷Be⁴⁺ + e⁻ → ⁷Li³⁺. Паттерн можно переиспользовать для других протон-избыточных ядер с малым Q.
  - **γ-распад** — возбуждённое ядро снимает энергию мощным γ-квантом, Z и A не меняются. Атомный аналог уже есть — `SpontaneousEmission` (электрон спускается по уровням). Ядерный γ-распад появится, когда введём возбуждённые состояния ядер — пока не моделируем.
  - **Спонтанное деление** — тяжёлое ядро (U, Pu, трансураниды) делится на два больших осколка + несколько нейтронов. Понадобится только если дойдём до моделирования атомной энергии / деления — далёкая перспектива.

  **Универсальная концепция t½** (период полураспада) — статистическое время, за которое половина популяции ядер распадается. Применим к **любому** виду из списка выше. Сейчас в `BetaPlusDecay` и `SpontaneousEmission` темп зашит как `chance(0.02f, ...)` — за тик ~2% шанс, что даёт ~50 тиков (~1 сек) среднего полураспада. Реальные t½ (минуты для ¹³N, миллиарды лет для ²³⁸U) сжаты до зрелищной секунды — сознательное огрубление; точная калибровка по изотопам — отдельный TODO когда наберём 5+ нестабильных видов.
- **Нейтринные реакции** — на каждой звёздной реакции в реальности рождаются нейтрино, которые уносят энергию.
- **Химические реакции с переходными состояниями (катализ)** — большинство реальных реакций идут через активационный барьер и могут ускоряться катализаторами.
- **Гравитация** — сейчас выключена (`gravityForce = 0` в `Entity.calculateForce`). Без неё нет планетообразования и аккреции.
- **Кулоновское отталкивание электронных оболочек** — упрощено до простой функции от количества электронов.

Когда дойдём до пребиотической химии и жизни, понадобятся ещё: ферменты, автокаталитические циклы, репликация молекул. Это уже отдельная эпоха симуляции.

## Игровой цикл (`composeApp/.../world/`)

`World` владеет:

- `entities` (`SnapshotStateList<Entity<*>>`)
- палитрой элементов
- корневым `Environment` (радиус 10000)
- `_pendingRequests: MutableList<ReactionRequest>` — буфер запросов реакций
- `random: Random(seed=1L)` — единый сидированный RNG для всей симуляции
- `EntityGenerator`, `ChemicalReactionResolver`

Один tick (`tickMs = 16L`) внутри единственного `_scope.launch`:

1. **Step phase** — `entities.toList().forEach { it.step() }`. Каждая сущность синхронно делает свой ход; может дописать запрос в `_pendingRequests`.
2. **Resolve phase** — берётся снимок `_pendingRequests`, список очищается, каждый запрос проходит через `ChemicalReactionResolver.resolve`, результат применяется (`consumed.destroy()`, `spawn()`, `updateState()`).
3. `delay(tickMs)` — ждём следующий кадр.

Никаких `Mutex`, `Channel` или `suspend` в самой симуляции — tick единственный писатель мира.

## 🚧 Рефакторинг на единый tick — статус

**Главное переключение завершено.** Симуляция гоняется одним tick-loop'ом в `World` (`tickMs = 16L`); сущности — пассивные `step()`-функции; в домене нет ни `Mutex`, ни `delay`, ни `Channel`, ни `suspend`. Tick — единственный писатель мира.

### Что осталось

1. **SpatialGrid (`Grid2D.kt`)** — оживить и подключить к World. Перестраивается в начале tick'а; `getNeighbors()` идёт через него. Главный пункт по производительности.
2. **Drag&drop спавн (`App.kt`) + Space «выстрел» (`RightPanel.kt`) + WASD-импульс (`World.applyForceToEntity`)** — сейчас UI пишет в `entities`/`environment.children`/`state.value` напрямую, мимо tick'а. Race в наших масштабах редко стреляет (Wasm однопоточный), но архитектурно — pending-список с применением в фазе Cleanup финализирует «tick — единственный писатель».
3. **Тест на детерминированность.** `World.random` сидируется константой `_seed = 1L`, инфраструктура готова — осталось написать `SharedCommonTest.kt`, который запускает мир дважды от одинакового стартового состояния и сверяет последовательность реакций.

### Решено по дизайну (не задача)

- **`updateMyEnvironment` посреди step'а — by design.** Сущности шагают **последовательно**, не «по снимку мира на начало tick'а». Если `SpaceModule` в свой ход затащил реагент в свою среду — реагент в свой ход (если он позже в обходе) действует уже в новой среде. Возвращаемся только если `SpatialGrid` потребует «один срез на tick» или захотим параллелить step.

## Технические концерны (TODO / на будущее)

Зафиксировано, чтобы не забыть. Список по убыванию приоритета:

1. **`getNeighbors()` сейчас O(N²).**
   На каждом шаге каждой сущности фильтруется весь список соседей. Файл `Grid2D.kt` уже лежит — видимо, в процессе перехода на spatial hashing. Без него масштабирование закончится быстро. *(Совпадает с пунктом «SpatialGrid» в текущем рефакторинге.)*

2. **`weight()` у всех правил возвращает `0f`.**
   Резолвер из-за этого выбирает случайное правило среди подходящих. Задумка «приоритизация реакций по вероятности» пока не работает — её нужно либо запустить, либо честно признать, что выбор просто случайный.

3. **`Entity` дефолтно бросает `Not Supported` на методы среды.**
   Type-smell: не каждая Entity — среда, и интерфейс это знает. Лучше отдельный `EnvironmentProvider` интерфейс, реализуемый только теми, кто реально среда (`Star`, `SpaceModule`).

4. **`Element` как enum + большой `Details`-каталог.**
   Сейчас норм, но когда дойдёт до органики (десятки/сотни молекул), enum станет узким горлом. Держать в уме переход на `data class Element` + регистр.

5. **`EntityState` с `var`-полями, но используется как immutable через `copyWith`.**
   Одно или другое — иначе путает читателя. Лучше сделать поля `val`.

6. **Мелочи:**
   - `id = ""` у нескольких правил, `id = "PhotoIonization"` у двух разных правил (`PhotoIonization` и `StarEmission`).
   - Смешение стилей в `Element` enum: `PHOTON` (всекапс) vs `Proton` / `Star` / `Ni` (CamelCase / короткое).

7. **Пустой `server` модуль.**
   Если планируется мультиплеер/сохранения — это отдельный большой проект. Если нет — можно убрать из `settings.gradle.kts`, чтобы не мозолил.

## Стратегические заметки (геймплей)

1. **Определить «победу» / прогрессию.**
   Что для игрока — конец партии? Достижение жизни? Технологическое дерево? Колонизация одной звёздной системы? Это направит, какие механики достраивать, а какие — нет.

2. **Разнести симуляцию на эпохи.**
   Между «протон поймал электрон» и «возникла самовоспроизводящаяся молекула» в реальной вселенной 10⁹–10¹⁷ разницы по времени и масштабу — на одной модели всю цепочку не покрыть. Естественные эпохи: космогенез → нуклеосинтез в звёздах → планетарная химия → пребиотика → жизнь. Внутри одной эпохи модель остаётся простой; переходы между эпохами — отдельная игровая веха.

3. **«Реакторы» как игровая механика, обходящая медленные естественные процессы.**
   `SpaceModule` / `RecombinationModule` — уже шаг в эту сторону. Игрок строит «ускорители», которые сжимают невероятно медленные естественные процессы. Это рабочая формула в духе Factorio / Oxygen Not Included, её стоит развивать.
