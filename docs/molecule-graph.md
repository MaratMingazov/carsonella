# Молекулы: граф вместо enum — дизайн

> Статус: **черновик / проектирование**. Будущая эпоха, делаем НЕ сейчас.
> Миграция выбрана **параллельная**: графовые молекулы живут рядом с enum-молекулами, мигрируем постепенно (живой движок не ломаем).
>
> Краткая версия и стадии — в [`ROADMAP.md`](../ROADMAP.md) (секция «Молекулы: граф вместо enum»). Здесь — детальный разбор.

## 1. Проблема: монолитная enum-молекула

Сейчас молекула — одна константа `Element` с агрегатными свойствами и плоским списком распада:

```kotlin
H2O  to Details(type = Molecule, symbol = "H₂O", mass = 18f, e = 10, p = 10, n = 8)
H2   to Details(..., energyBondDissociation = 4.5f, dissociationElements = [HYDROGEN, HYDROGEN])
```

Доказательство тупика — прямо в каталоге:

```kotlin
C2_H6_O_ETHANOL         to Details(symbol = "C₂H₅OH",  mass=46, e=26, p=26, n=20)
C2_H6_O_DIMETHYL_ETHER  to Details(symbol = "CH₃OCH₃", mass=46, e=26, p=26, n=20)
```

Две разные молекулы — одинаковые агрегаты (C₂H₆O). Различие только в **связности** (C–O–H vs C–O–C), которой модель не хранит.

Следствия:
- **Комбинаторный взрыв.** Каждая молекула = константа. Изомеры уже плодят дубли; органик — миллионы; ДНК — полимер на миллиарды звеньев. Перечислить нельзя.
- **Нет графа атомов/связей.** Без топологии не смоделировать: какая связь рвётся, реакционные центры, катализ, полимеризацию, репликацию. `dissociationElements` топологию не несёт (и задан только у H₂).
- **Реакции захардкожены попарно** (`AtomPlusAtomToMolecule(H,H,H2)`). Нужны шаблоны над функциональными группами, а не enum на каждую пару.

## 2. Целевая модель: молекула = граф

Атомы — узлы, связи — рёбра. Сущность держит **структуру, а не идентичность**; формула/масса/символ вычисляются из графа.

Структуры вписаны в этот кодбейс — переиспользуем `Element` как «изотоп» узла и `Vec2D` для геометрии:

```kotlin
data class AtomNode(
    val localId: Int,
    val isotope: Element,      // HYDROGEN, OXYGEN_16 — отсюда p/n/mass узла
    val charge: Int = 0,       // формальный заряд / электроны (стык с ионизацией)
    val offset: Vec2D = Vec2D(0f, 0f),  // положение внутри молекулы (для рендера)
)

data class Bond(
    val a: Int,                // localId узла
    val b: Int,                // localId узла
    val order: Int = 1,        // 1/2/3 — одинарная/двойная/тройная
)

data class MoleculeGraph(
    val nodes: List<AtomNode>,
    val bonds: List<Bond>,
) {
    fun mass(): Float          // сумма масс узлов — больше НЕ хранится в Details
    fun protons(): Int
    fun formula(): String      // "C2H6O" — из графа
    fun canonical(): String    // канон. форма для сравнения «это вода?» и реестра
}
```

Этанол и диметиловый эфир станут **двумя разными графами с одной формулой** — естественно, без дублей-констант.

## 3. Интеграция с движком (главная развилка)

Весь движок завязан на `EntityState.element: Element` — рендер, силы (`calculateForce` читает `element.details.mass/radius`), матчинг реакций. Граф ломает это допущение. Два пути:

### 3a. Параллельно — ВЫБРАНО
- В `MoleculeState` добавляем `graph: MoleculeGraph?`.
- Агрегаты молекулы (масса/радиус/символ) считаются из графа, когда он есть; иначе — из `Details` (старые enum-молекулы).
- Атомы/субатомы — без изменений на enum.
- Старые enum-молекулы (`H2`, `O2`, …) живут рядом; мигрируем по одной.

Плюсы: маленькие шаги, живой движок не ломаем. Минус: какое-то время два представления сосуществуют — нужен чистый мост (см. §4).

### 3b. Заменить целиком — отложено
Ввести `Species = Atomic(isotope, electrons) | Molecular(graph)` вместо `element` во всём движке. Чище в пределе (единая модель), но это большой разовый рефактор рендера/сил/реакций/save-load. Не сейчас.

## 4. Мост enum ↔ граф

Чтобы графовая молекула «ехала» по текущему движку, нужна тонкая прослойка, дающая агрегаты:

```
MoleculeGraph → mass / radius / symbol / formula
```

Молекула-сущность в `step()` двигается и считает силы по этим агрегатам (как сейчас по `Details`), а реакции — уже по графу (см. §6). На этапе миграции мост умеет работать и со старым `Details`, и с новым `graph`.

## 5. Трудные подзадачи (назвать честно сразу)

1. **Каноникализация графа** — чтобы понять «этот граф = вода», сравнивать молекулы и вести реестр известных. Это граф-изоморфизм (каноническая нумерация). Решаемо, но нетривиально; первый шаг — наивно (для малых графов). Подробно — см. §5.1.
2. **Валентность/связи** — можно ли двум атомам связаться и сколько связей. Проще — таблица валентностей (H:1, O:2, N:3, C:4). Строже — октет/электроны (стыкуется с ионизацией: свободные «слоты» = валентные электроны).
3. **Реакции = шаблоны над подграфами** (SMARTS-подобно). Старт с минимума: «образовать связь» / «разорвать связь». Дальше — функциональные группы (этерификация и т.п.).
4. **Геометрия/рендер.** Сейчас молекула — один кружок. Граф можно рисовать связанными атомами (по `offset`). Опционально, но наглядно и мотивирует.
5. **Save/Load.** DTO-слой уже есть и графу на руку: молекула сериализуется как `nodes`+`bonds` в JSON естественно. При миграции `EntityDto` молекулы получает опциональный блок графа.
6. **Детерминизм.** Каноникализация и применение шаблонов должны быть детерминированы (важно для теста воспроизводимости из README TODO).

## 5.1. Каноникализация графа — зачем и как

### Зачем
Один и тот же граф можно записать по-разному — узлы пронумерованы произвольно:

```
вода, нумерация А:   0:H — 1:O — 2:H
вода, нумерация Б:   0:O — 1:H , 0:O — 2:H
```

Это один H₂O, но в памяти — разные списки `nodes`/`bonds`. Нужна **каноническая форма** —
детерминированная строка/ключ, одинаковая у молекулы при любой перенумерации. Тогда:
- «этот граф = вода?» → сравнить канонические строки;
- молекула как ключ `Map` / подсчёт в `summary`;
- дедупликация (один и тот же продукт, собранный двумя путями);
- различение изомеров (этанол ≠ диметиловый эфир при одной формуле C₂H₆O).

Формально это **изоморфизм графов** с метками (тип атома на узле, кратность на ребре);
каноникализация = **каноническая нумерация** узлов.

### Подход 1 — наивный (перебор)
Перебрать все перестановки номеров узлов, для каждой построить строку, взять
лексикографически минимальную.
- ✅ точно и просто; ❌ O(n!). Для ≤ ~10 атомов — нормально. **Берём на старте.**

### Подход 2 — алгоритм Моргана (1965, Chemical Abstracts)
Вычислить «расширенную связность» (EC) каждого атома и упорядочить по ней:
1. раунд 0: EC = степень (число связей);
2. раунд k+1: EC = сумма EC соседей;
3. повторять, пока растёт число различных значений EC; затем взять старт по макс. EC,
   обойти граф, разрывая ничьи по типу атома / заряду / кратностям → каноническая нумерация.

**Пример (различение изомеров C₂H₆O), только тяжёлые атомы:**
```
этанол:           C — C — O      степени: C1 C2 O1  → «двойной» узел = углерод
диметиловый эфир: C — O — C      степени: C1 O2 C1  → «двойной» узел = кислород
```
Канонические строки получаются разные → молекулы различаются, хотя формула одна.

### Тонкость и продакшн-ориентиры
Чистый Морган спотыкается на симметрии (ничьи); реальные алгоритмы добавляют инварианты
и итеративное уточнение разбиения. Ориентиры, если упрёмся: **canonical SMILES** (CANGEN),
**InChI** (слоистый идентификатор), **nauty** (общая канон. форма графа).

**Как выглядит каноническая строка в реальном стандарте (InChI).** Те же изомеры C₂H₆O:
```
этанол:            InChI=1S/C2H6O/c1-2-3/h3H,2H2,1H3
диметиловый эфир:  InChI=1S/C2H6O/c1-3-2/h1-2H3
```
Формула одна (`C2H6O`), но **слой связности `c…` различается** (`c1-2-3` vs `c1-3-2`) — ровно
этим InChI и разделяет изомеры. Плюс `InChIKey` — 27-символьный хеш строки (фикс. длина,
удобен как ключ): у этанола `LFQSCWFLJHTTHZ-UHFFFAOYSA-N`.

InChI — золотой стандарт каноникализации (внутри Морган-подобная нормализация + каноническое
мечение), но это C-код, не KMP/Wasm-дружелюбный и избыточный для наших крошечных молекул —
держим как **ориентир**, не зависимость. Эталонная реализация (IUPAC/InChI Trust, C, MIT):
<https://github.com/IUPAC-InChI/InChI>.

### Полимеры — бесплатный случай
Для ДНК/белков весь граф каноникализировать не надо: идентичность = **последовательность**
(`ATGC…` / аминокислоты), а строка сама по себе уже канонична.

### План по стадиям
- **Стадия 0–1:** наивная каноникализация (перебор / сортировка по простым инвариантам).
- **Стадия 2:** Морган-подобный EC + разрыв ничьих, когда молекулы подрастут.
- **Стадия 3:** идентичность полимера = последовательность, граф-каноникализация не нужна.

## 5.2. MOL/SDF — референс-формат сериализации графа

**Chemical table files (CTfile)** — семейство текстовых форматов от MDL для записи молекул;
по сути это стандартный способ сериализовать молекулярный граф. Главные:
- **MOL** (`.mol`) — одна молекула: блок атомов + блок связей;
- **SDF** (`.sdf`) — много молекул (записи MOL через `$$$$`) + произвольные поля данных;
- **RXN/RDF** — реакции.

Структура MOL (V2000): заголовок → counts-строка (сколько атомов/связей) → atom block
(символ элемента + координаты X Y Z) → bond block (`атом1 атом2 кратность`, 1/2/3/4) →
properties (`M CHG` заряды, `M ISO` изотопы) → `M END`. Пример — вода:
```
  3  2  0  0  0  0  0  0  0  0999 V2000
    0.0000    0.0000    0.0000 O   0  0
    0.9572    0.0000    0.0000 H   0  0
   -0.2399    0.9266    0.0000 H   0  0
  1  2  1  0
  1  3  1  0
M  END
```

**Прямое соответствие нашей модели:**

| MOL | наша модель |
|---|---|
| atom block (символ + координаты) | `List<AtomNode>` (`isotope` + `offset`) |
| bond block (`атом1 атом2 кратность`) | `List<Bond>` (`a`, `b`, `order`) |
| `M CHG` | `AtomNode.charge` |

Значение для нас: подтверждает, что модель «атомы-узлы + связи-рёбра» — индустриальный
стандарт, и даёт готовую раскладку полей (координаты/кратность/заряд/изотоп) для `AtomNode`/`Bond`.
Потенциальный путь **импорта/экспорта** молекул и вход для InChI (граф → MOL → каноническая
строка, чтобы сверять нашу каноникализацию). Сейчас — **ориентир**, не зависимость; наш
основной формат сохранения — JSON-DTO (см. §5, save/load).

## 5.3. Идентификация: канонический ключ → смысл

Каноникализация (§5.1) отвечает только «равны ли две молекулы» и «видел ли я это раньше».
Сам ключ **смысла не несёт**: `C2H6O/c1-2-3/...` не знает, что это «спирт». Понять, *что это* —
отдельный слой.

**Смена мышления: большинство молекул безымянны — и это нормально.** В enum-мире каждая
молекула обязана быть заранее названа. В граф-мире симуляция рождает миллионы молекул;
называть все нельзя и не нужно. Молекула работает как граф и без имени. Имя/смысл — про
**курируемое подмножество** интересных молекул.

### Лестница смыслов (от дешёвого к богатому)
- **a) Брутто-формула** — из графа, бесплатно, всегда (`C₂H₆O`). Грубая классификация.
- **b) Канонический ключ** — точная идентичность, различает изомеры. «Отпечаток», не имя.
- **c) Функциональные группы (классификация по подструктуре)** ⭐ — искать в графе характерные
  фрагменты: `–OH` → спирт; `–COOH` → кислота; пептидная связь `–C(=O)–N–` → пептид/белок;
  цепочка нуклеотидов → нуклеиновая кислота. Даёт **класс** («какой-то спирт») без точного
  совпадения и без реестра. Технически — то же сопоставление подграфов, что и шаблонные
  реакции (§6): один движок — две пользы.
- **d) Точное имя — реестр известных молекул** (см. §7).

### Полимеры — идентификация иначе
Реестр по ключу для ДНК/белков не работает (нельзя занести каждый геном). Идентичность =
**последовательность**; «что это» = **узнавание мотивов/доменов** в последовательности
(известный рибозим, белковый домен) — то же «по фрагментам», но над строкой.

Итого «что это?» = формула (бесплатно) + класс по функциональным группам (масштабируемо) +
точное имя из реестра (для курируемых). Канонический ключ — мост к последнему.

## 6. Реакции как шаблоны

Вместо `AtomPlusAtomToMolecule(H, H, H2)` — правило-шаблон над графом:

- **Образование связи:** два фрагмента, у каждого есть свободный валентный слот совместимого типа → добавить ребро, слить графы. (H· + ·H → H–H; ·OH + H· → H₂O.)
- **Разрыв связи (диссоциация):** выбрать ребро, разорвать → два подграфа. Заменяет `dissociationElements` — топология определяет, какая связь рвётся и что получится.
- **Перенос группы / замещение:** шаблон «найти подграф A–B, заменить на A–C» — основа органической химии (потом).

Старые попарные правила переносятся на граф-движок по одному и затем удаляются.

## 7. Реестр известных молекул и крючок прогрессии

Точное имя молекулы (уровень «d» из §5.3) даёт **реестр** — курируемая таблица
`канонический ключ → известная молекула`:

```kotlin
data class KnownMolecule(
    val canonicalKey: String,
    val name: String,        // "Вода"
    val symbol: String,      // "H₂O"
    val description: String,
    // опц.: особое поведение, реальные свойства
)
object MoleculeRegistry {
    private val byKey: Map<String, KnownMolecule>      // курируем вручную
    fun identify(graph: MoleculeGraph): KnownMolecule? // null = неизвестная
}
```

Логика при образовании молекулы:
```
ключ = graph.canonical()
известная = registry.identify(graph)
есть  → показываем имя, применяем особое поведение
нет   → неизвестная: формула «C₂H₆O (неизв. изомер)», но полноценный граф
```

- **Сидируется той же каноникализацией:** строим граф воды → ключ → запись. Один алгоритм
  и для посева реестра, и для поиска → согласованно.
- **Реестр — наследник `Element` enum + `Details`:** знание (имя воды, свойства) сохраняем,
  а требование перечислить ВСЕ молекулы — теряем. Неизвестным запись не нужна.
- **Крючок прогрессии:** первое появление известной молекулы (вода, аминокислота, АТФ,
  нуклеотид) — это **событие-веха** («открыта вода!», «первая аминокислота!»). Реестр — не
  только подпись на канве, а механика открытий и переходов между эпохами (ср. «Стратегические
  вопросы геймплея» в [`ROADMAP.md`](../ROADMAP.md)).

## 8. Связь с ионизационным рефакторингом

Ортогональны, но складываются. Ионизация даёт **атом = (изотоп + электроны в state)** — ровно это и есть узел графа (`AtomNode.isotope + charge`). Один принцип на трёх уровнях:

```
ион → изотоп-узел → молекула-граф → полимер-последовательность
```

## 9. Стадии амбиции (от малого к большому)

Делаем по нарастающей; каждая стадия — отдельный заход.

- **Стадия 0 — чистое ядро графа.** `MoleculeGraph/AtomNode/Bond` + `formula/mass/canonical`, отдельный модуль, юнит-тесты. Движок не трогаем. Низкий риск, фундамент.
- **Стадия 1 — PoC на текущих молекулах.** Перевыразить H₂/O₂/H₂O/CH₄ как графы через мост (§4), поведение то же. Доказать модель на малом.
- **Стадия 2 — общая химия малых молекул.** Валентности + шаблоны связей (§6): рождаются НОВЫЕ молекулы CHNO, которых нет в списке (включая изомеры). Рендер связанных атомов.
- **Стадия 3 — полимеры / жизнь.** Последовательности (ДНК/белки) + матричные операции (репликация/транскрипция/трансляция) как правила высшего уровня. Отдельный «жизненный» слой.

## 10. Инкрементальный план (маленькими шагами)

1. Стадия 0: ядро графа + тесты (commonMain, изолировано).
2. Мост `MoleculeGraph → агрегаты`; `MoleculeState.graph` (опциональный, параллельно enum).
3. Граф-реакции: воспроизвести H+H→H₂ и распад через шаблоны; затем убрать попарные правила.
4. Рендер связанных атомов (опц.).
5. Валентности + рождение новых молекул (стадия 2).
6. Полимеры (стадия 3) — отдельной эпохой.

## 11. Открытые вопросы

- Каноникализация: подтвердить план «наивная → Морган → последовательности» (см. §5.1); на какой границе по числу атомов переключаться с перебора на EC?
- Валентность: фиксированная таблица или вывод из электронов (связь с ионизацией)?
- Геометрия: считать `offset` физикой (мини-силы между узлами) или раскладывать схематично?
- Где провести границу «атом-сущность» ↔ «узел графа»: молекула владеет узлами как данными (узлы — НЕ отдельные Entity) — принять это явно.
- Идентификация (§5.3): когда вводить классификацию по функциональным группам (классы «спирт/кислота») — сразу или после реестра точных имён? Что считать «известной» молекулой = веха прогрессии (§7)?
