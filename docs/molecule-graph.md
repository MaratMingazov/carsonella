# Молекулы: граф вместо enum — дизайн

> Миграция выбрана **параллельная** (§3a): графовые молекулы живут рядом с enum, движок мигрируем по шагам.
> Детальный разбор и принятые решения — ниже (§2, §3b, §5.1, §6, §8, §11). Краткая версия — в [`ROADMAP.md`](../ROADMAP.md).

## Прогресс (актуальное состояние)

**Сделано:**
- **Стадия 0** — ядро графа `chemistry/graph/MoleculeGraph.kt` (`AtomNode`/`Bond`/`MoleculeGraph`;
  `mass`/`protons`/`formula`/`formulaPretty`/`canonical`) + тесты.
- **§3b (инкрементально)** — `Species = Elemental(Element) | Molecular(MoleculeGraph)` вместо `element`
  (шов `EntityState.element` работает для Elemental); агрегаты через `Species.mass/radius/protons/displaySymbol`;
  `EntityGenerator` создаёт сущности из `Species`; молекула = `Molecular(graph)`, граф non-null.
- **`Element.valence()`** — вычисляется из Z по октету (CHNO верно; Z>18 → 0).
- **Шаг 3a — образование связи**: `CovalentBondFormation` (два нейтральных лёгких атома → двухатомная
  `Molecular`, одинарная связь). `H+H→H₂` работает в приложении.
- **Структурный рендер**: `EntityRenderer.drawMolecule` + `MoleculeLayout`; атомы молекулы рисуются тем же
  «светящимся шаром» (`drawAtomOrb`), что и одиночный атом, только компактнее.
- **Фикс краша реагентов-молекул**: абстрактный `AtomReactionRule` (гейтит субъект на `Elemental`) — на него
  переведены ~20 атомных/ядерных правил; `PhotoDissociation` дормантный (enum-молекулы не спавнятся).
  Реорг пакетов: `rules/atom_rules/` (+`CovalentBondFormation`), `rules/molecule_rules/`. Регрессионный тест.
- **Шаг 3b — рост молекулы**: `MoleculeReactionRule` (субъект — `Molecular`) + `MoleculeGrowth`
  (атом+молекула / молекула+молекула через `MoleculeGraph.merge` + `freeSlots`/`firstFreeSlotNode`).
  **Вода собирается**: O+H→·OH (3a), затем ·OH+H→H₂O (3b); также H₂O₂.
- **Каталог энергий связи** `BondEnergy` — средние энергии по ТИПУ связи (пара элементов + кратность), эВ,
  ~20 записей на CHNO; примитив рядом с `Element.details` (см. §6).
- **Энергия образования связи**: `CovalentBondFormation`/`MoleculeGrowth` излучают фотон `E(связи)`
  (радиационная ассоциация — сохранение энергии; фотон дальше фото-ионизирует/диссоциирует соседей).
- **Шаг 3c — усиление связи**: `BondStrengthening` (само-реакция, `reagents.size != 1`) + `MoleculeGraph.strengthenBond`/
  `strengthenableBonds`; изолированная O–O → O=O, N₂ → N≡N; фотон на прирост `E(k+1)−E(k)`. `weight()` пока 0.

**Предстоит:**
- **Веса «рост vs усиление»**: адаптировать `weight()` под энергетический выбор (max выигрыш `E` из `BondEnergy`) —
  уберёт бесконечные цепочки густого кислорода (O–O→O=O предпочтётся даже при партнёре рядом). См. §6.
- **Граф-диссоциация**: оживить `PhotoDissociation` на графе — продукты из топологии, порог = `BondEnergy`
  (решение вар. A уже в §6).
- **Замыкание колец** (§6, §11): внутримолекулярные связи → кольца/листы (бензол, графен); нужна геометрия узлов.
- **VSEPR-геометрия** (§11): форма молекулы (вода «уголком», а не линией) через стерическое число + лоун-пары.
- ионы-реагенты; реестр (§7, опц.); зум/LOD рендера; богатая таксономия `Species` (выпилить `ElementType`,
  см. §3b); полимеры (стадия 3).

## 1. Проблема: монолитная enum-молекула

Сейчас молекула — одна константа `Element` с агрегатными свойствами и плоским списком распада:

```kotlin
H2O  to Details(type = Molecule, symbol = "H₂O", mass = 18f, e = 10, p = 10, n = 8)
H2   to Details(..., energyBondDissociation = 4.5f, dissociationElements = [HYDROGEN, HYDROGEN])
```

Доказательство тупика — прямо в каталоге:

```kotlin
C2_H6_O_ETHANOL         to Details(symbol = "C₂H₅OH",  mass=46, e=26, p=26, n=20)
C2_H6_O_DIMETHYL_ETHER  to Details(symbol = "CH₃OCH₃", mass=46, e=26, p=26, n=20)
```

Две разные молекулы — одинаковые агрегаты (C₂H₆O). Различие только в **связности** (C–O–H vs C–O–C), которой модель не хранит.

Следствия:
- **Комбинаторный взрыв.** Каждая молекула = константа. Изомеры уже плодят дубли; органик — миллионы; ДНК — полимер на миллиарды звеньев. Перечислить нельзя.
- **Нет графа атомов/связей.** Без топологии не смоделировать: какая связь рвётся, реакционные центры, катализ, полимеризацию, репликацию. `dissociationElements` топологию не несёт (и задан только у H₂).
- **Реакции захардкожены попарно** (`AtomPlusAtomToMolecule(H,H,H2)`). Нужны шаблоны над функциональными группами, а не enum на каждую пару.

## 2. Целевая модель: молекула = граф

Атомы — узлы, связи — рёбра. Сущность держит **структуру, а не идентичность**; формула/масса/символ вычисляются из графа.

Типы живут в коде: `AtomNode` / `Bond` / `MoleculeGraph` — `shared/.../chemistry/graph/MoleculeGraph.kt`
(смотри там; определения в доке не дублируем — разъедутся). Узел переиспользует `Element` как изотоп;
агрегаты `mass()` / `protons()` / `formula()` / `canonical()` считаются из графа, а не хранятся.

Этанол и диметиловый эфир — **два разных графа с одной формулой** C₂H₆O, естественно, без дублей-констант.

**Решения Стадии 0** (чего из кода не видно — почему так):
- **Узел = только изотоп.** Электроны в графе НЕ хранятся: это динамическое состояние сущности
  (`state.electrons`), а связи в графе и ЕСТЬ общие электроны (см. §8). Поэтому per-atom `charge`/`offset`
  на узле нет (вернём `charge`, если понадобится формальный заряд; `offset` — на стадии рендера).
- **Масса = Σ (p+n) узлов** — поле `Details.mass` убрано (всегда равнялось p+n).
- **`Bond.order` без дефолта** — кратность указываем явно, чтобы не получить молча одинарную связь из-за
  забытого значения. Гард `1..3` (выше — лишь экзотика переходных металлов).

## 3. Интеграция с движком (главная развилка)

Весь движок завязан на `EntityState.element: Element` — рендер, силы (`calculateForce` читает `element.details.mass/radius`), матчинг реакций. Граф ломает это допущение. Два пути:

### 3a. Параллельно — ВЫБРАНО
- В `MoleculeState` добавляем `graph: MoleculeGraph?`.
- Агрегаты молекулы (масса/радиус/символ) считаются из графа, когда он есть; иначе — из `Details` (старые enum-молекулы).
- Атомы/субатомы — без изменений на enum.
- Старые enum-молекулы (`H2`, `O2`, …) живут рядом; мигрируем по одной.

Плюсы: маленькие шаги, живой движок не ломаем. Минус: какое-то время два представления сосуществуют — нужен чистый мост (см. §4).

### 3b. Заменить `element` на `Species` — В РАБОТЕ (инкрементально)
Идентичность сущности = sealed `Species` вместо голого `Element`. Делаем НЕ большим разовым рефактором,
а по шагам, через шов: у `EntityState` остаётся геттер `element`, работающий для не-молекул, поэтому
весь не-молекулярный код (ядерные реакции, рендер, save-load) не трогается.

**Сделано:** `Species = Elemental(Element) | Molecular(MoleculeGraph)` (граф у молекулы non-null by
construction); агрегаты (масса/радиус/протоны/символ) считаются по `Species`. Дальше — молекулы
создаются как `Molecular(graph)` через фабрику, и enum-молекулы (`H2O`/`O2`/…) + член `ElementType.Molecule`
уходят, когда молекулы полностью переедут на граф.

**Будущий шаг — богатая таксономия `Species` (выпилить `ElementType`):** позже расщепить `Elemental`
на физические виды, чтобы «вид материи» стал type-safe (исчерпывающий `when`) и тег `Details.type` стал не нужен:
```kotlin
sealed interface Species {
    data class Elementary(/* частица */) : Species  // фотон / электрон / протон / нейтрон / позитрон
    data class Nuclear(val element: Element) : Species  // атомы (изотоп — носитель p/n/radius/symbol)
    data class Molecular(val graph: MoleculeGraph) : Species
}
```
Нюансы того захода:
- классификация `Element → вид` не исчезает — переезжает в фабрику `Species.of(element)` / `when(element)`;
  места, держащие голый `Element` (`Element.symbol()`, `canGainElectron`, термоионизация), должны получать `Species`;
- сам `Element` остаётся — как носитель `p/n/radius/symbol`/ядерных реакций внутри `Nuclear`/`Elementary`;
- **Star/модули — не материя, а контейнеры среды** (`IEnvironment` с детьми) → в таксономию материи не
  входят; для них классификация (или отдельный тип) остаётся, так что `ElementType` уйдёт не весь разом.

## 4. Мост enum ↔ граф

Чтобы графовая молекула «ехала» по текущему движку, нужна тонкая прослойка, дающая агрегаты:

```
MoleculeGraph → mass / radius / symbol / formula
```

Молекула-сущность в `step()` двигается и считает силы по этим агрегатам (как сейчас по `Details`), а реакции — уже по графу (см. §6). На этапе миграции мост умеет работать и со старым `Details`, и с новым `graph`.

## 5. Трудные подзадачи (назвать честно сразу)

1. **Каноникализация графа** — чтобы понять «этот граф = вода», сравнивать молекулы и вести реестр известных. Это граф-изоморфизм (каноническая нумерация). Решаемо, но нетривиально; первый шаг — наивно (для малых графов). Подробно — см. §5.1.
2. **Валентность/связи** — можно ли двум атомам связаться и сколько связей. Проще — таблица валентностей (H:1, O:2, N:3, C:4). Строже — октет/электроны (стыкуется с ионизацией: свободные «слоты» = валентные электроны).
3. **Реакции = шаблоны над подграфами** (SMARTS-подобно). Старт с минимума: «образовать связь» / «разорвать связь». Дальше — функциональные группы (этерификация и т.п.).
4. **Геометрия/рендер.** Сейчас молекула — один кружок. Граф можно рисовать связанными атомами (по `offset`). Опционально, но наглядно и мотивирует.
5. **Save/Load.** DTO-слой уже есть и графу на руку: молекула сериализуется как `nodes`+`bonds` в JSON естественно. При миграции `EntityDto` молекулы получает опциональный блок графа.
6. **Детерминизм.** Каноникализация и применение шаблонов должны быть детерминированы (важно для теста воспроизводимости из README TODO).

## 5.1. Каноникализация графа — зачем и как

### Зачем
Один и тот же граф можно записать по-разному — узлы пронумерованы произвольно:

```
вода, нумерация А:   0:H — 1:O — 2:H
вода, нумерация Б:   0:O — 1:H , 0:O — 2:H
```

Это один H₂O, но в памяти — разные списки `nodes`/`bonds`. Нужна **каноническая форма** —
детерминированная строка/ключ, одинаковая у молекулы при любой перенумерации. Тогда:
- «этот граф = вода?» → сравнить канонические строки;
- молекула как ключ `Map` / подсчёт в `summary`;
- дедупликация (один и тот же продукт, собранный двумя путями);
- различение изомеров (этанол ≠ диметиловый эфир при одной формуле C₂H₆O).

Формально это **изоморфизм графов** с метками (тип атома на узле, кратность на ребре);
каноникализация = **каноническая нумерация** узлов.

### Подход 1 — наивный (перебор)
Перебрать все перестановки номеров узлов, для каждой построить строку, взять
лексикографически минимальную.
- ✅ точно и просто; ❌ O(n!). Для ≤ ~10 атомов — нормально. **Берём на старте.**

### Подход 2 — алгоритм Моргана (1965, Chemical Abstracts)
Вычислить «расширенную связность» (EC) каждого атома и упорядочить по ней:
1. раунд 0: EC = степень (число связей);
2. раунд k+1: EC = сумма EC соседей;
3. повторять, пока растёт число различных значений EC; затем взять старт по макс. EC,
   обойти граф, разрывая ничьи по типу атома / заряду / кратностям → каноническая нумерация.

**Пример (различение изомеров C₂H₆O), только тяжёлые атомы:**
```
этанол:           C — C — O      степени: C1 C2 O1  → «двойной» узел = углерод
диметиловый эфир: C — O — C      степени: C1 O2 C1  → «двойной» узел = кислород
```
Канонические строки получаются разные → молекулы различаются, хотя формула одна.

### Тонкость и продакшн-ориентиры
Чистый Морган спотыкается на симметрии (ничьи); реальные алгоритмы добавляют инварианты
и итеративное уточнение разбиения. Ориентиры, если упрёмся: **canonical SMILES** (CANGEN),
**InChI** (слоистый идентификатор), **nauty** (общая канон. форма графа).

**Как выглядит каноническая строка в реальном стандарте (InChI).** Те же изомеры C₂H₆O:
```
этанол:            InChI=1S/C2H6O/c1-2-3/h3H,2H2,1H3
диметиловый эфир:  InChI=1S/C2H6O/c1-3-2/h1-2H3
```
Формула одна (`C2H6O`), но **слой связности `c…` различается** (`c1-2-3` vs `c1-3-2`) — ровно
этим InChI и разделяет изомеры. Плюс `InChIKey` — 27-символьный хеш строки (фикс. длина,
удобен как ключ): у этанола `LFQSCWFLJHTTHZ-UHFFFAOYSA-N`.

InChI — золотой стандарт каноникализации (внутри Морган-подобная нормализация + каноническое
мечение), но это C-код, не KMP/Wasm-дружелюбный и избыточный для наших крошечных молекул —
держим как **ориентир**, не зависимость. Эталонная реализация (IUPAC/InChI Trust, C, MIT):
<https://github.com/IUPAC-InChI/InChI>.

### Полимеры — бесплатный случай
Для ДНК/белков весь граф каноникализировать не надо: идентичность = **последовательность**
(`ATGC…` / аминокислоты), а строка сама по себе уже канонична.

### План по стадиям
- **Стадия 0–1:** наивная каноникализация перебором — **сделано** (гард ≤ 9 узлов; ключ из изотопов + нормализованных рёбер с кратностью; различает изомеры этанол/эфир).
- **Стадия 2:** Морган-подобный EC + разрыв ничьих, когда молекулы подрастут.
- **Стадия 3:** идентичность полимера = последовательность, граф-каноникализация не нужна.

## 5.2. MOL/SDF — референс-формат сериализации графа

**Chemical table files (CTfile)** — семейство текстовых форматов от MDL для записи молекул;
по сути это стандартный способ сериализовать молекулярный граф. Главные:
- **MOL** (`.mol`) — одна молекула: блок атомов + блок связей;
- **SDF** (`.sdf`) — много молекул (записи MOL через `$$$$`) + произвольные поля данных;
- **RXN/RDF** — реакции.

Структура MOL (V2000): заголовок → counts-строка (сколько атомов/связей) → atom block
(символ элемента + координаты X Y Z) → bond block (`атом1 атом2 кратность`, 1/2/3/4) →
properties (`M CHG` заряды, `M ISO` изотопы) → `M END`. Пример — вода:
```
  3  2  0  0  0  0  0  0  0  0999 V2000
    0.0000    0.0000    0.0000 O   0  0
    0.9572    0.0000    0.0000 H   0  0
   -0.2399    0.9266    0.0000 H   0  0
  1  2  1  0
  1  3  1  0
M  END
```

**Прямое соответствие нашей модели:**

| MOL | наша модель |
|---|---|
| atom block (символ + координаты) | `List<AtomNode>` (`isotope` + `offset`) |
| bond block (`атом1 атом2 кратность`) | `List<Bond>` (`a`, `b`, `order`) |
| `M CHG` | `AtomNode.charge` |

Значение для нас: подтверждает, что модель «атомы-узлы + связи-рёбра» — индустриальный
стандарт, и даёт готовую раскладку полей (координаты/кратность/заряд/изотоп) для `AtomNode`/`Bond`.
Потенциальный путь **импорта/экспорта** молекул и вход для InChI (граф → MOL → каноническая
строка, чтобы сверять нашу каноникализацию). Сейчас — **ориентир**, не зависимость; наш
основной формат сохранения — JSON-DTO (см. §5, save/load).

## 5.3. Идентификация: канонический ключ → смысл

Каноникализация (§5.1) отвечает только «равны ли две молекулы» и «видел ли я это раньше».
Сам ключ **смысла не несёт**: `C2H6O/c1-2-3/...` не знает, что это «спирт». Понять, *что это* —
отдельный слой.

**Смена мышления: большинство молекул безымянны — и это нормально.** В enum-мире каждая
молекула обязана быть заранее названа. В граф-мире симуляция рождает миллионы молекул;
называть все нельзя и не нужно. Молекула работает как граф и без имени. Имя/смысл — про
**курируемое подмножество** интересных молекул.

### Лестница смыслов (от дешёвого к богатому)
- **a) Брутто-формула** — из графа, бесплатно, всегда (`C₂H₆O`). Грубая классификация.
- **b) Канонический ключ** — точная идентичность, различает изомеры. «Отпечаток», не имя.
- **c) Функциональные группы (классификация по подструктуре)** ⭐ — искать в графе характерные
  фрагменты: `–OH` → спирт; `–COOH` → кислота; пептидная связь `–C(=O)–N–` → пептид/белок;
  цепочка нуклеотидов → нуклеиновая кислота. Даёт **класс** («какой-то спирт») без точного
  совпадения и без реестра. Технически — то же сопоставление подграфов, что и шаблонные
  реакции (§6): один движок — две пользы.
- **d) Точное имя — реестр известных молекул** (см. §7).

### Полимеры — идентификация иначе
Реестр по ключу для ДНК/белков не работает (нельзя занести каждый геном). Идентичность =
**последовательность**; «что это» = **узнавание мотивов/доменов** в последовательности
(известный рибозим, белковый домен) — то же «по фрагментам», но над строкой.

Итого «что это?» = формула (бесплатно) + класс по функциональным группам (масштабируемо) +
точное имя из реестра (для курируемых). Канонический ключ — мост к последнему.

## 6. Реакции как шаблоны

Вместо `AtomPlusAtomToMolecule(H, H, H2)` — правило-шаблон над графом:

- **Образование связи:** два фрагмента, у каждого есть свободный валентный слот совместимого типа → добавить ребро, слить графы. (H· + ·H → H–H; ·OH + H· → H₂O.)
- **Разрыв связи (диссоциация):** выбрать ребро, разорвать → два подграфа. Заменяет `dissociationElements` — топология определяет, какая связь рвётся и что получится.
- **Замыкание кольца (внутримолекулярная связь):** связать два ненасыщенных атома ОДНОЙ молекулы (обычно концы цепи со свободными слотами) → цепь становится кольцом (бензол, циклогексан, сахара). Отличается от роста тем, что связывает атомы внутри одной молекулы, а не две разные сущности. **Зачем нужно:** без него чистый углерод растёт линейными цепями до бесконечности, тогда как в реальности он собирается в кольца/листы/каркасы (графен, алмаз, фуллерены). Требует геометрии узлов (понять, что концы сблизились) и предпочтения размера кольца (5–6-звенные выгодны, 3–4 напряжены) — см. §11. Пока НЕ реализовано; в химии жизни проблема снимается терминацией водородом (концы `–CH₃` закрывают слоты → молекулы конечны).
- **Перенос группы / замещение:** шаблон «найти подграф A–B, заменить на A–C» — основа органической химии (потом).

Старые попарные правила переносятся на граф-движок по одному и затем удаляются.

### Бухгалтерия электронов в реакциях (решение)
Реакция обязана **сохранять сумму электронов** — это свойство правила, не типа узла.
- **Образование:** `molecule.electrons = Σ(electrons реагентов)` (см. §8).
- **Разрыв (диссоциация):** сумма осколков = электроны молекулы; но КАК делить — выбор. В мире — два
  механизма: **гомолитический** (пара делится поровну → нейтральные радикалы) и **гетеролитический**
  (более электроотрицательный осколок забирает оба электрона → ионы). Если молекула уже была ионом,
  дефицит ложится на наименее электроотрицательный осколок (станет катионом). Данные для выбора есть —
  энергии ионизации по Z (`energyLevelsTable()`).
- **Текущий `PhotoDissociation` неполон:** даёт каждому осколку нейтральный счёт (`details.p`) — сохраняет
  электроны лишь для НЕЙТРАЛЬНОЙ молекулы. Для ионов нужно обобщить правилом выше (будущая задача).

### Граф-диссоциация: продукты из топологии, порог — каталог энергий связи (решение)
Зеркало образования связи (3b): рост **добавляет** ребро и сливает графы, распад **убирает** ребро и
разбивает граф. `PhotoDissociation` сейчас дормантный (enum-молекулы не спавнятся) — вот как он оживёт на графе:

- **Продукты — из топологии, не из `dissociationElements`.** Удаляем ребро → ищем связные компоненты:
  фрагмент из одного узла → `Elemental(изотоп)` (атом), из ≥2 узлов → `Molecular(подграф)` (с переиндексацией,
  как `merge` в 3b — общая графовая хирургия, `split` это её зеркало). Пример: H₂O, рвём O–H → ·OH + H·.
  Краевой случай — **кольцо**: если после удаления ребра граф остался связным (ребро в цикле), это не распад,
  а раскрытие кольца; проверка связности это ловит. Колец пока нет.
- **Порог — энергия связи по ТИПУ, не по молекуле.** Вместо `energyBondDissociation` на каждую молекулу —
  каталог по типу связи: ключ = неупорядоченная пара элементов + кратность. Для всей CHNO это ~20 записей
  (H–H, O–H, C–H, C–C, C=O, O=O, N≡N…), и они **переиспользуются всеми молекулами** (метанол/глюкоза/ДНК —
  те же C–C, C–O, O–H). Размер не растёт с числом молекул. Двухатомный частный случай совпадает со старым
  полем: `H2.energyBondDissociation = 4.5` — это и есть энергия связи H–H.
  - **РЕШЕНО — вариант A: прямая эмпирическая таблица** (`bondEnergy(a, b, order): Float`, backed by
    `Map<BondKey, Float>`, сеём лениво по мере появления типов). Это **примитивная** физическая константа —
    как `p/n/radius`/энергии ионизации в `Element.details`; граф убирает *составной* хардкод (продукты, масса),
    но не примитивы. Каталог связей живёт рядом с `Element` как «таблица Менделеева для связей».
  - Отвергнуто: (B) приближение `base × order` — врёт (C–C = O–H); (C) формула Полинга по χ элементов —
    меньше данных, но это приближение и константы (χ, гомоядерные E) всё равно хранятся. Для ~20 значений
    прямая таблица проще и точнее.
- **Какую связь рвём:** доступная энергия (фотон + накопленная) ≥ порога → связь рвётся; среди доступных
  берём слабейшую (рвётся первой) либо случайную из доступных.
- **Электроны осколков:** гомолитически (нейтральная молекула → нейтральные радикалы) — дефолт; ионы/
  гетеролитика — позже (см. бухгалтерию электронов выше).

### Образование связи и её кратность (решение + будущие шаги)
**Сделано (Шаг 3a):** `CovalentBondFormation` — вычисляемое правило (не таблица пар): два близких
нейтральных лёгких атома со свободными слотами (`Element.valence() > 0`) → двухатомная `Molecular`.
Кратность стартует **одинарной** (`order = 1`) — это всегда корректно (H₂, C–C, C–H, O–H…).

**Кратность НЕ вычисляем заранее** через `min(valence)` — это ловушка: для углерода дало бы C₂
четверную (и невалидно >3, и неверно — C–C одинарная). Вместо этого кратные связи **эмёрджентны**:
- **3b — рост:** молекула с незакрытым слотом притягивает ещё атом/молекулу (атом+молекула → вода).
- **3c — усиление:** если два связанных атома остаются ненасыщенными и **нового партнёра рядом нет**,
  связь усиливается 1→2→3. Так O₂ становится O=O, N₂ → N≡N, а углерод (рядом много партнёров)
  расходует слоты на разные одинарные связи (цепи).

**Граничные случаи для 3c — не забыть:**
- **Потолок 3** (главные группы выше тройной не идут; четверные+ — только переходные металлы, мы их
  не связываем). Усиление останавливается на 3; `Bond.require(1..3)` страхует. Атом может остаться с
  открытым слотом (радикал) — это нормально, насыщать всё не обязаны.
- **Сначала новый партнёр, усиление — крайняя мера** (иначе углерод неверно «доусиливался» бы вместо цепей).
- **Октет/валентность — лишь приближение.** Настоящий порядок связи = заполнение молекулярных орбиталей
  `(связывающие − разрыхляющие)/2`, а не счёт по октету. Поэтому эвристика будет врать на экзотике:
  реальный C₂ — двойная (порядок 2), а наше насыщение дало бы тройную; парамагнетизм O₂ октет вообще не
  объясняет. Для симулятора жизни это приемлемо (C₂ не встречается) — МО-теорию не тянем, ограничения принимаем.

### Озон (O₃) — явно НЕ покрывается моделью (и почему)
Центральному кислороду в озоне нужна **валентность 3** (три связи, формальный заряд `+`), а `Element.valence()`
(`Entity.kt`) фиксирована на **нейтральный** атом и считается из Z по октету (O → 2). Per-atom электронов/заряда
в графе нет (§8: заряд — свойство МОЛЕКУЛЫ, не атома), поэтому переменную валентность «O⁺ делает 3 связи»
**вывести неоткуда** — модель этого не выражает. Плюс реальный O₃ делокализован (порядок каждой связи ~1.5,
уголок, формальные заряды), а целочисленные кратности + октет этого не несут: наивный рост дал бы **линейный
бирадикал `•O–O–O•`** (структурно неверный), а не уголковый озон. И механизм иной — реальный озон собирается
`O + O₂` **только при подводе энергии** (УФ/разряд рвёт O₂ на атомы), а не эмёрджентно из валентного роста.

Чтобы покрыть озон, понадобилась бы **валентность из per-atom электронного баланса** (формальный заряд/электроны
на узле → «O⁺ = 3 связи») — отложенный слой (§8 «опциональный `AtomNode.charge`, но не физическая правда»; §11
«валентность: фиксированная таблица или вывод из электронов?»). До тех пор озон — **курируемый спецслучай**
(реестр §7 / отдельное правило `O + O₂` с порогом по энергии), а не результат обычной сборки графа.

## 7. Реестр известных молекул и крючок прогрессии

> Статус: **необязательный курируемый слой, НЕ ядро.** Симуляция работает на анонимных графах и без него
> (молекулы опознаются структурно: формула/канон). Реализуем в будущем — когда понадобятся имена,
> особое поведение и вехи-открытия. Большинство молекул так и останутся безымянными — это нормально (§5.3).

Точное имя молекулы (уровень «d» из §5.3) даёт **реестр** — курируемая таблица
`канонический ключ → известная молекула`:

```kotlin
data class KnownMolecule(
    val canonicalKey: String,
    val name: String,        // "Вода"
    val symbol: String,      // "H₂O"
    val description: String,
    // опц.: особое поведение, реальные свойства
)
object MoleculeRegistry {
    private val byKey: Map<String, KnownMolecule>      // курируем вручную
    fun identify(graph: MoleculeGraph): KnownMolecule? // null = неизвестная
}
```

Логика при образовании молекулы:
```
ключ = graph.canonical()
известная = registry.identify(graph)
есть  → показываем имя, применяем особое поведение
нет   → неизвестная: формула «C₂H₆O (неизв. изомер)», но полноценный граф
```

- **Сидируется той же каноникализацией:** строим граф воды → ключ → запись. Один алгоритм
  и для посева реестра, и для поиска → согласованно.
- **Реестр — наследник `Element` enum + `Details`:** знание (имя воды, свойства) сохраняем,
  а требование перечислить ВСЕ молекулы — теряем. Неизвестным запись не нужна.
- **Крючок прогрессии:** первое появление известной молекулы (вода, аминокислота, АТФ,
  нуклеотид) — это **событие-веха** («открыта вода!», «первая аминокислота!»). Реестр — не
  только подпись на канве, а механика открытий и переходов между эпохами (ср. «Стратегические
  вопросы геймплея» в [`ROADMAP.md`](../ROADMAP.md)).

## 8. Связь с ионизационным рефакторингом

Ортогональны, но складываются. Ионизация даёт **атом = (изотоп + электроны в state)**; узел графа — это
изотоп (ядро), а электроны остаются на уровне сущности. Один принцип на трёх уровнях:

```
ион → изотоп-узел → молекула-граф → полимер-последовательность
```

«Перестать перечислять составное enum-константами; представлять составное через части + правила.»

### Электроны и заряд молекулы (решение)
- **Заряд — свойство МОЛЕКУЛЫ, не атома.** При ионизации электрон уходит с делокализованной молекулярной
  орбитали — «какой атом потерял» физически обычно не определено. Поэтому per-atom заряда в графе нет;
  заряд считаем целиком у молекулы. (Формальный заряд на атоме — лишь бухгалтерская условность; если
  понадобится, это опциональный `AtomNode.charge`, но не физическая правда.)
- **Число электронов — в `MoleculeState.electrons`** (как `AtomState.electrons` у атома). В графе электронов
  нет; их наличие закодировано связями (связь = общая пара).
- **Сохранение при образовании:** `molecule.electrons = Σ(electrons реагентов)`. Так из O⁺ + 2 H рождается
  ион воды H₂O⁺.
- **Заряд считается** как `graph.protons() − state.electrons` — та же формула, что у атома (`p − electrons`).
- **Канон заряд не включает:** `canonical()` — про структуру (как изотоп не меняется от ионизации);
  H₂O и H₂O⁺ имеют один структурный ключ, различает их заряд в состоянии.

## 9. Стадии амбиции (от малого к большому)

Делаем по нарастающей; каждая стадия — отдельный заход.

- **Стадия 0 — чистое ядро графа.** `MoleculeGraph/AtomNode/Bond` + `formula/mass/canonical`, отдельный модуль, юнит-тесты. Движок не трогаем. Низкий риск, фундамент.
- **Стадия 1 — PoC на текущих молекулах.** Перевыразить H₂/O₂/H₂O/CH₄ как графы через мост (§4), поведение то же. Доказать модель на малом.
- **Стадия 2 — общая химия малых молекул.** Валентности + шаблоны связей (§6): рождаются НОВЫЕ молекулы CHNO, которых нет в списке (включая изомеры). Рендер связанных атомов.
- **Стадия 3 — полимеры / жизнь.** Последовательности (ДНК/белки) + матричные операции (репликация/транскрипция/трансляция) как правила высшего уровня. Отдельный «жизненный» слой.

## 10. Инкрементальный план (маленькими шагами)

1. Стадия 0: ядро графа + тесты (commonMain, изолировано).
2. Мост `MoleculeGraph → агрегаты`; `MoleculeState.graph` (опциональный, параллельно enum).
3. Граф-реакции: воспроизвести H+H→H₂ и распад через шаблоны; затем убрать попарные правила.
4. Рендер связанных атомов (опц.).
5. Валентности + рождение новых молекул (стадия 2).
6. Полимеры (стадия 3) — отдельной эпохой.

## 11. Открытые вопросы

- Каноникализация: план «наивная → Морган → последовательности» (см. §5.1). Стадия 0 РЕШЕНА перебором с гардом **≤ 9 узлов** (9! ≈ 363k — ещё мгновенно; 10! на порядок дороже). Открыто: точная граница и переход на Морган-подобный EC, когда молекулы перерастут гард.
- Валентность: фиксированная таблица или вывод из электронов (связь с ионизацией)?
- Геометрия: считать `offset` физикой (мини-силы между узлами) или раскладывать схематично? Два РАЗНЫХ применения:
  (1) **замыкание колец** (§6) — нужна пространственная раскладка узлов, чтобы понять, что концы цепи сблизились;
  (2) **форма молекулы** (угол связей). Важная тонкость: простое отталкивание связанных атомов даёт формы «без
  лоун-пар» — 2 связи → 180° (линия), 3 → 120° (плоскость), 4 → тетраэдр. Поэтому вода вышла бы ЛИНЕЙНОЙ, а не
  уголком: изгиб H–O–H (104.5°) создают неподелённые пары кислорода (VSEPR), которых в графе нет (электроны —
  счётчик на молекуле, не по атомам/доменам). Чтобы получить уголок/пирамиду, нужна **VSEPR-эвристика**: угол по
  стерическому числу (связи + лоун-пары), где лоун-пары вычисляются из валентных электронов минус связывающие.
  Отдельная фича — «бесплатно из геометрии для колец» угол воды НЕ починится.
- Где провести границу «атом-сущность» ↔ «узел графа»: молекула владеет узлами как данными (узлы — НЕ отдельные Entity) — принять это явно.
- Идентификация (§5.3): когда вводить классификацию по функциональным группам (классы «спирт/кислота») — сразу или после реестра точных имён? Что считать «известной» молекулой = веха прогрессии (§7)?
